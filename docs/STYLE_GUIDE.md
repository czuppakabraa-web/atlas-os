# STYLE GUIDE

Version: 1.0

---

# Purpose

This document defines the visual identity of Atlas OS.

Every screen, component and interaction must follow these rules.

The objective is consistency.

Users should immediately recognize Atlas without seeing the logo.

---

# Design Philosophy

Atlas is not a fitness app.

Atlas is not a social network.

Atlas is not a game.

Atlas is an operating system.

Everything should feel like professional software.

---

- Blade Runner 2049
- Ghost in the Shell
- NASA Mission Control
- Modern aircraft cockpit
- Bloomberg Terminal
- Professional monitoring dashboards

Secondary

- Tesla UI
- Rivian UI

- Instagram

- Facebook

- MyFitnessPal
- Material Design templates

Technical

# User Experience

The interface should disappear.

The user should focus on information.

Not on decorations.

---

# Theme

Default

Dark

Future

Light mode may be added.

Dark mode is always the primary design.

---

# Colors

Primary

Electric Blue

Purpose

Interactive elements

Selection

Highlights

Links

---

Background

Almost Black

Purpose

Reduce eye fatigue

Increase focus

---

Cards

Dark Gray

Purpose

Separate information

Never use bright cards.

---

Success

Green

Only for positive confirmation.

---

Warning

Orange

Only when user attention is required.

---

Danger

Red

Reserved for errors.

Never use red as decoration.

---

# Typography

Large titles

Thin

Elegant

Lots of whitespace

Body

Readable

Simple

Metrics

Large

Bold

Easy to scan

Captions

Small

Muted

Never use decorative fonts.

---

# Spacing

Whitespace is part of the design.

Prefer larger spacing over visual clutter.

The interface should breathe.

---

# Cards

Cards group information.

Cards never contain unnecessary decorations.

Cards never contain gradients.

Cards should appear as monitoring panels.

---

# Tables

Nutrition

Weight

Reports

Analytics

should primarily use tables instead of cards.

Atlas is a data system.

Tables communicate data faster.

---

# Charts

Charts should answer questions.

Not decorate screens.

Allowed

Line

Bar

Area

Avoid

Pie charts

3D charts

Animated charts

---

# Icons

Use only when improving readability.

Avoid icon overload.

Prefer text whenever possible.

---

# Animations

Animations support understanding.

They never exist for entertainment.

Duration

Fast

Purpose

State changes

Navigation

Feedback

Never animate everything.

---

# Navigation

Navigation should be predictable.

Maximum depth

Three levels.

The user should never feel lost.

---

# Dashboard

The dashboard is a system overview.

It answers one question.

"What is happening today?"

---

# Module Philosophy

Every module answers one question.

Examples

Dashboard

How am I today?

Nutrition

What have I eaten?

Weight

How is my body changing?

Activity

What have I done?

Analytics

Why is this happening?

Atlas AI

What should I do next?

---

# Data Density

Atlas prefers information density over decoration.

One screen should provide meaningful information without becoming cluttered.

---

# Empty States

Empty states should explain what is missing.

Never simply display:

"No data"

Instead explain:

What data is expected.

How to add it.

Why it matters.

---

# Loading States

Always use skeletons or subtle indicators.

Never block the application unnecessarily.

---

# Error States

Errors should explain

What happened

Why

How to fix it

Never expose technical messages.

---

# Accessibility

Support

Large fonts

Screen readers

High contrast

Large touch targets

Readable spacing

Accessibility is not optional.

---

# Future Design Direction

The application should increasingly resemble:

Mission Control

instead of

Traditional Mobile App.

---

# Golden Rule

Every screen should answer:

"What is the user trying to learn?"

If an element does not help answer that question,

it should be removed.
